# OctoAcme — Project Management Processes

OctoAcme runs work through a lightweight, iterative delivery workflow centered on a prioritized backlog and a project board with clear columns (Backlog, Ready, In Progress, In Review, QA, Done). Initiatives start with a Project One-pager to capture the problem, goals, success metrics, and stakeholders, then move into planning (kickoff, backlog creation, estimates, and Definition of Done) and execution. Pull requests are kept small when possible, must link to the related issue, include acceptance criteria, and are gated by CI and at least one reviewer approval before merging. Releases follow a checklist-driven pipeline (staging smoke tests, automated production deploy, and post-deploy verifications) with rollback plans for incidents.

Roles and responsibilities are explicit: Product Managers define outcomes, prioritize the backlog, and measure success; Project Managers coordinate schedules, risks, and stakeholder communications; Developers implement features, tests, and docs; QA validates acceptance criteria and runs manual or automated testing; stakeholders provide approvals and domain inputs. These personas are used to assign ownership for artifacts such as the Project One-pager, release notes, risk register, and action items coming out of retrospectives or incidents, ensuring clear accountability across the lifecycle.

Communication is structured and regular: daily standups for progress and blockers, a weekly delivery sync to surface progress and risks, PM–PdM weekly alignment, and monthly stakeholder updates. The docs emphasize a single source of truth (project README or release doc) and provide templates for weekly status and incident communications so updates are consistent and actionable. Escalation paths are defined and security incidents follow the security runbook and on-call notification process.

Quality assurance is both preventive and verification-focused: unit and integration tests for new logic, end-to-end smoke tests for critical flows, security scanning in CI, and manual QA for feature acceptance when needed. The process requires CI passing before review, a Definition of Done for work readiness, and explicit acceptance criteria on backlog items and PRs. Continuous improvement is embedded via regular retrospectives that produce prioritized action items tracked back into the backlog or project board so quality and process gaps are closed iteratively.
